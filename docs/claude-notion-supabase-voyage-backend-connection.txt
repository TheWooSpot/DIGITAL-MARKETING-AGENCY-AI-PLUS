Great question — and it's the right one to ask before we convert, because once it's a number field everywhere, reversing it is painful.

Text vs Number for Service ID — here's the honest breakdown:

When text is more powerful:

Prefixes and suffixes. If you ever want IDs like SVC-101, SOC-802, or V2-101 — text handles that, number can't.
Leading zeros. 0101 as a display convention — number strips them.
API string matching. Some payment and CRM systems (including some Stripe metadata fields) expect IDs as strings and do exact string matching — "101" not 101. Text is safer for cross-system lookups.
Composite IDs. If you later need 101-A or 101-EN for variants — text supports that instantly.
Notion formulas referencing ID as text. Formula fields that concatenate ID into a URL or label need text — "stripe.com/product/" + Service_ID works with text, not with a number field.
When number is more powerful:

Sort order. Exactly your use case — 101 before 201 before 1001, guaranteed.
Math. Range queries, averages, greater-than filters — none of which you need for an ID.
Supabase integer foreign keys. If Supabase layer1_service_catalog uses an integer primary key, number aligns cleanly.
My actual recommendation given your full stack:
